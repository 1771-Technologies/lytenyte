## What Are Expressions

An expression is a string of text that produces a value. In its simplest form, an expression can
be a math equation such as `1 + 1`, which evaluates to the value `2`.

Expressions can represent more complex calculations in a concise form. For example, the expression
`sum(Sales Revenue) / sum(Sales Quantity)` computes the average sale revenue per unit.

LyteNyte Grid's expression capabilities allow developers to define a custom expression language.
Users can then use this language to describe the computation they want to perform. Two common
use cases include:

- **[Expression Filters](/docs/expression-filters):** Complex filtering rules written concisely as expressions.
- **[Expression Fields](/docs/expression-fields):** Dynamically computed cell values based on expressions.

## LyteNyte Grid's Expression Engine

Expressions are just text input. The term `sum(Sales Revenue) / sum(Sales Quantity)` has no meaning
on its own. LyteNyte Grid provides an expression engine that evaluates these expressions to produce
a concrete value.

The expression engine takes the text representation and performs three steps:

1. **Tokenization:** Breaks the expression into its component parts. This step defines the building
   blocks (tokens) of the expression language.
2. **Parse:** Uses the tokens from the previous step to create a structured representation of the
   expression. This representation is a data structure that defines the relationships between parts
   of the expression so the engine can evaluate it correctly. Programmers refer to this as the
   Abstract Syntax Tree (AST).
3. **Evaluate:** Computes the expression by resolving any required values. This step takes an
   expression and produces a final value. The grid can then use this value to filter rows, compute
   cell values, or support other use cases in your application.

### The Evaluator

LyteNyte Grid's expression engine is exposed as an exported class called `Evaluator`. At its core, the `Evaluator`
acts as a calculator that can evaluate basic math expressions.

The code below shows the simplest use case:

```ts
import { Evaluator } from "@1771technologies/lytenyte-pro/expressions";

const result = new Evaluator().run("1 + 1"); // 2
```

LyteNyte Grid's `Evaluator` supports a plugin system that extends its capabilities with more advanced
expression features. The library provides a standard set of plugins that enable a JavaScript-like
expression language that developers can use out of the box.

The demo below shows this more advanced expression language. The standard expression language
supports identifiers (`age`), functions (`fullName()`), and interpolated strings.

### Expression Plugins

Expression plugins extend the `Evaluator` class with additional capabilities. Pass an array of plugins
when creating an instance of the `Evaluator` to enable specific features. For example:

```ts
import { Evaluator, standardPlugins, stringsPlugin } from "@1771technologies/lytenyte-pro/expressions";

// Evaluator with standard expression language support
const standardEvaluator = new Evaluator(standardPlugins);

// Evaluator with string support (+ basic math operations)
const stringEvaluator = new Evaluator([stringsPlugin]);
```

### Expression Context

For an expression such as `age + 21`, the `Evaluator` needs a way to resolve the value of the
identifier `age`. The `Evaluator.run` method accepts a context object as a second argument.
This object provides the runtime values for identifiers used in the expression.

```ts
const context = {
  age: 23,
  user: { firstName: "John", lastName: "Smith" },
  fullName: () => context.user.firstName + " " + context.user.lastName,
};

evaluator.run("age * 2", context); // 46
evaluator.run("user.firstName", context); // "John"
evaluator.run("fullName()", context); // "John Smith"
```

The context can include any value type, such as primitives, objects, arrays, and functions.
Expressions can call functions defined in the context directly. This mechanism allows you to expose
domain-specific data and operations to expression authors.

### Standard Expression Language

The base `Evaluator` (without plugins) supports arithmetic operators: `+`, `-`, `*`, `/`, `%`,
and `**` (exponentiation).

The `standardPlugins` set turns the base evaluator into a JavaScript-like expression language.
The table below lists each plugin, the syntax it adds, and an example.

| Plugin              | What it adds                                  | Example                                  |
| ------------------- | --------------------------------------------- | ---------------------------------------- |
| `stringsPlugin`     | String literals and template literals         | `"hello"`, `'world'`, `` `Hi ${name}` `` |
| `booleansPlugin`    | `true`, `false`, `null`, `undefined` literals | `status == true`                         |
| `comparisonPlugin`  | `==` `!=` `<` `>` `<=` `>=`                   | `age >= 18`                              |
| `logicalPlugin`     | `&&` `\|\|` `??` and unary `!`                | `a && b \|\| c`                          |
| `membershipPlugin`  | `in` and `not in`                             | `"key" in obj`                           |
| `ternaryPlugin`     | `condition ? then : else`                     | `x > 0 ? "pos" : "neg"`                  |
| `collectionsPlugin` | Array and object literals, spread (`...`)     | `[1, 2, ...rest]`                        |
| `arrowsPlugin`      | Arrow functions (single and multi-parameter)  | `x => x * 2`, `(a, b) => a + b`          |
| `pipePlugin`        | Pipe operator `\|>`                           | `items \|> filter(x => x > 0)`           |
| `accessPlugin`      | `.prop`, `[computed]`, `?.`, function calls   | `user.name`, `obj?.city`                 |

#### Literals

Numbers support decimal, hexadecimal (`0xFF`), octal (`0o77`), binary (`0b1010`), scientific
notation (`1e10`), and underscore separators (`1_000_000`).

Strings support single quotes, double quotes, and standard escape sequences (`\n`, `\t`, `\uXXXX`,
`\xHH`). Template literals use backticks and support embedded expressions:

```
`${user.firstName} scored ${points * multiplier} points`
```

#### Operators

Operators follow standard precedence, from lowest to highest:

| Operators                       | Description                             |
| ------------------------------- | --------------------------------------- |
| `?:`                            | Ternary conditional                     |
| `\|>`                           | Pipe                                    |
| `\|\|`                          | Logical OR                              |
| `&&`                            | Logical AND                             |
| `??`                            | Nullish coalescing                      |
| `==` `!=`                       | Equality                                |
| `<` `>` `<=` `>=` `in` `not in` | Relational and membership               |
| `+` `-`                         | Addition and subtraction                |
| `*` `/` `%`                     | Multiplication, division, modulo        |
| `**`                            | Exponentiation (right-associative)      |
| `!` `-` `+`                     | Unary NOT, negation, identity           |
| `.` `[…]` `?.` `()`             | Member access, computed, optional, call |

#### Property Access and Function Calls

Access nested values using `.` (dot notation) or `[computed]` (bracket notation). The `?.`
operator returns `undefined` when the left side is `null` or `undefined` instead of throwing:

```ts
user.address.city; // throws if address is null
user.address?.city; // returns undefined if address is null
data["dynamic-key"]; // computed bracket access
```

Call any function in the context or a method on a value:

```ts
fullName(); // calls fullName from the context
"hello".toUpperCase(); // method call (this binding preserved)
items.filter((x) => x > 0); // method with arrow function argument
```

#### Arrow Functions

Arrow functions are first-class values. Use them as callbacks for array methods or any
function in the context that accepts a function argument:

```ts
items.map((x) => x * 2);
items.filter((x, i) => x > 0 && i < 10);
```

Single-parameter arrows do not require parentheses. Multi-parameter arrows do.

:::note

Arrow functions in the expression language are not closures. Identifiers in the surrounding
scope are not captured. The expression language does not support scoping rules or variable
declarations.

:::

#### Pipe Operator

The `|>` operator passes the left-hand value as the **last argument** to the right-hand
function or call expression. Chains read left-to-right without deep nesting:

```ts
// Without pipe
round(multiply(price, 1.1), 2)

// With pipe
price |> multiply(1.1) |> round(2)
```

If the right side is a bare identifier, the input is passed as the sole argument:
`value |> format`.

#### Logical Short-Circuit and Nullish Coalescing

The `&&` and `||` operators short-circuit, so the right side evaluates only when needed. The `??` operator
returns the right side only when the left side is `null` or `undefined` (unlike `||`, which also
triggers on `0` or `""`):

```ts
user && user.name; // safe: user.name only evaluated if user is truthy
count || "none"; // returns "none" if count is 0 (falsy)
label ?? "default"; // returns "default" only if label is null/undefined
```

## Expressions As A Standalone Language

The expression engine has no dependency on LyteNyte Grid. The `Evaluator`, `standardPlugins`, and
the plugin system are general-purpose. You can import and use them in any JavaScript or
TypeScript project. The grid integration (filters, computed fields, the `ExpressionEditor`
component) builds on the same public API.

Some examples of what you can build with expressions outside of a grid context:

- **Rule engines:** Evaluate user-defined conditions against application state, such as access control rules, notification triggers, or business logic that non-developers configure.
- **Formula inputs:** Let users enter calculations in a form field, for example a pricing configurator where a sales rep enters `basePrice * quantity * (1 - discount)`.
- **Template rendering:** Use template literals and the pipe operator to build lightweight text templates, generating messages or documents from a context object without a full templating library.
- **Configuration DSLs:** Store expressions as strings in a database or config file and evaluate them at runtime, keeping computed logic out of code and making it changeable without deployment.
- **Workflow automation:** Evaluate step conditions in a visual workflow builder, where each node's transition condition is an expression authored by the user.
