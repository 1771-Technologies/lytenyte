## Creating Expression Filters

An expression filter evaluates to a boolean value (`true` or `false`). LyteNyte Grid evaluates
the expression for each row and keeps the row when the result is `true`.

To improve the user experience for filter expressions, use column names instead of column IDs.
For example, a user writes `Gender == "Male"` instead of `row.customerGender == "Male"`.
Two mechanisms enable this:

1. **`createResolvedIdentifierPlugin`**: Intercepts identifier nodes in the AST and rewrites
   them as function calls. This ensures `Gender` resolves from the context at evaluation time
   instead of being treated as a plain key.
2. **The context object**: Maps each column display name to a function that reads the column value
   from the current row using `computeField`.

The demo below preloads the expression `Gender == "Male" && Quantity > 10`. Edit the expression to filter
across any column:

The demo includes the complete expression filter implementation. Expression filters require only a few
lines of code and let users create complex filtering logic in a concise and flexible way. The following
sections cover the implementation details.

### Setting Up the Evaluator

Start by creating an `Evaluator` instance. Create the evaluator outside your component or in a memo
so it remains a stable reference. Use `standardPlugins` and `createResolvedIdentifierPlugin` to support
advanced expression capabilities:

```ts
import {
  createResolvedIdentifierPlugin,
  Evaluator,
  standardPlugins,
} from "@1771technologies/lytenyte-pro/expressions";

const evaluator = new Evaluator(
  standardPlugins.concat([
    createResolvedIdentifierPlugin({
      identifiers: columns.map((x) => x.name ?? x.id),
      args: ["row"],
    }),
  ]),
);
```

### Building the Filter Function

One way to filter rows in LyteNyte Grid is to provide a [filter function](/docs/client-source-filtering).
To use an expression as a filter function:

1. Create a context object that maps column names to functions that return the column value for a given row:

   ```ts
   import { computeField } from "@1771technologies/lytenyte-pro";
   import type { RowLeaf } from "@1771technologies/lytenyte-pro/types";

   const columnsContext = useMemo(
     () =>
       Object.fromEntries(
         columns.map((col) => [col.name ?? col.id, (row: RowLeaf) => computeField(col.field ?? col.id, row)]),
       ),
     [],
   );
   ```

2. Pre-parse the expression into an AST. This avoids tokenizing the expression on every predicate evaluation.
   Set `undefinedIdentifierFallback` to `true` so partially typed expressions return `true` instead of throwing
   and error:

   ```ts
   const filter = useMemo(() => {
     if (!committed) return null;
     try {
       const ast = evaluator.ast(committed);

       return (row: RowLeaf<GridSpec["data"]>) =>
         !!evaluator.run(ast, { row, ...columnsContext }, { undefinedIdentifierFallback: true });
     } catch {
       return null; // invalid expression, show all rows
     }
   }, [committed, columnsContext]);
   ```

3. Pass the filter function to `useClientDataSource`:

   ```ts
   const ds = useClientDataSource<GridSpec>({ data, filter });
   ```

### Debouncing the Input

Filtering the grid on every keystroke is possible, but it is unnecessarily expensive. Debounce
the filter expression input so the grid updates after the user pauses typing.

Keep two state values:

- `value`: the value shown in the editor.
- `committed`: the value used to compute the filter.

The `committed` state trails `value` while the user types and updates after the user pauses.

```ts
const [value, setValue] = useState('Gender == "Male" && Quantity > 10');
const [committed, setCommitted] = useState(value);
const debounceRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const onChange = (s: string) => {
  if (debounceRef.current) clearTimeout(debounceRef.current);
  setValue(s);
  debounceRef.current = setTimeout(() => setCommitted(s), 200);
};
```

Pass `value` to `ExpressionEditor.Root` and `committed` to the filter memo. The editor updates
immediately, while the grid waits for the user to pause typing.

### Wiring Up the Completion Provider

Pass a completion provider to `ExpressionEditor.Root` to enable autocomplete for column names.
Use `createCompletionProvider` to create the default provider. Provide a `Record<displayName, id>` map
so the provider does not require access to the full context object:

```ts
const provider = useMemo(
  () => createCompletionProvider(Object.fromEntries(columns.map((col) => [col.name ?? col.id, col.id]))),
  [],
);
```

## Server-Side Expression Filters

The evaluator can compute expression values on the client when all required data is available. If your data
lives on the server, serialize the expression AST and evaluate it on the server instead:

```ts
const ast = evaluator.ast(committed);
const serialized = JSON.stringify(ast); // send in your DataRequest
```

The AST is fully serializable as long as your plugins do not introduce custom node types that
cannot be converted to JSON. `standardPlugins` and `createResolvedIdentifierPlugin` produce only
serializable nodes.
