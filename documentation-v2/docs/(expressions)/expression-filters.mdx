## How Expression Filters Work

An expression filter is an expression that evaluates to a boolean. The evaluator runs it against
each row and includes only the rows where the result is truthy.

Column names become identifiers in the expression. A user writes `Gender == "Male"` rather than
`row.customerGender == "Male"`. Two things make this work:

1. **`createResolvedIdentifierPlugin`**: intercepts identifier nodes in the AST and rewrites
   them as function calls, so `Gender` is resolved from the context at evaluation time rather than
   looked up as a plain key.
2. **The context object**: maps each column display name to a function that reads the column's
   value from the current row using `computeField`.

The demo below preloads the expression `Gender == "Male" && Quantity > 10`. Edit it to filter
across any column:

## Setting Up the Evaluator

Build the evaluator once outside your component. Register `createResolvedIdentifierPlugin` after
`standardPlugins`, passing the column display names as `identifiers` and `"row"` as the context
key the plugin should forward to each column getter:

```ts
import {
  createResolvedIdentifierPlugin,
  Evaluator,
  standardPlugins,
} from "@1771technologies/lytenyte-pro/expressions";

const evaluator = new Evaluator(
  standardPlugins.concat([
    createResolvedIdentifierPlugin({
      identifiers: columns.map((x) => x.name ?? x.id),
      args: ["row"],
    }),
  ]),
);
```

## Building the Filter Function

Construct the context inside `useMemo` so it is stable across re-renders. Each entry maps a
column's display name to a function that reads its value from a row via `computeField`:

```ts
import { computeField } from "@1771technologies/lytenyte-pro";
import type { RowLeaf } from "@1771technologies/lytenyte-pro/types";

const columnsContext = useMemo(
  () =>
    Object.fromEntries(
      columns.map((col) => [col.name ?? col.id, (row: RowLeaf) => computeField(col.field ?? col.id, row)]),
    ),
  [],
);
```

Pre-parse the expression into an AST so the filter does not re-parse on every row. Pass
`undefinedIdentifierFallback: true` so that a partially-typed expression (with identifiers the
evaluator does not yet recognise) returns `true` instead of throwing, keeping rows visible while
the user is still typing:

```ts
const filter = useMemo(() => {
  if (!committed) return null;
  try {
    const ast = evaluator.ast(committed); // parse once

    return (row: RowLeaf<GridSpec["data"]>) =>
      !!evaluator.run(ast, { row, ...columnsContext }, { undefinedIdentifierFallback: true });
  } catch {
    return null; // invalid expression, show all rows
  }
}, [committed, columnsContext]);
```

Pass the filter to `useClientDataSource`:

```ts
const ds = useClientDataSource<GridSpec>({ data, filter });
```

## Debouncing the Input

Run the filter only on committed input, not on every keystroke. Keep two state values: `value`
(what the editor displays) and `committed` (what the filter memo depends on). Debounce the commit
so the grid re-filters after the user pauses typing, not on every character:

```ts
const [value, setValue] = useState('Gender == "Male" && Quantity > 10');
const [committed, setCommitted] = useState(value);
const debounceRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const onChange = (s: string) => {
  if (debounceRef.current) clearTimeout(debounceRef.current);
  setValue(s);
  debounceRef.current = setTimeout(() => setCommitted(s), 200);
};
```

Pass `value` to `ExpressionEditor.Root` and `committed` to the filter memo. The editor updates
immediately while the grid waits for the user to pause.

## Wiring Up the Completion Provider

Pass a `createCompletionProvider` to `ExpressionEditor.Root` so the user gets autocomplete for
column names. Supply a `Record<displayName, id>` map so the provider knows the display names
without needing access to the full context object:

```ts
const provider = useMemo(
  () => createCompletionProvider(Object.fromEntries(columns.map((col) => [col.name ?? col.id, col.id]))),
  [],
);
```

## Server-Side Expression Filters

Expression evaluation on the server is straightforward because the AST is serializable. Instead
of evaluating client-side, serialize the AST and send it to your server:

```ts
const ast = evaluator.ast(committed);
const serialized = JSON.stringify(ast); // send in your DataRequest
```

The AST is fully serializable as long as your plugins do not introduce custom node types that
cannot be converted to JSON. Standard plugins and `createResolvedIdentifierPlugin` produce only
serializable nodes.
